1. Introduction

1.1. Purpose
This Software Requirements Specification (SRS) describes the functional and non-functional requirements for Version 1.0 of the Automated Faculty Student Management System (AFSMS). It covers how AFSMS will manage curricula, student records, and document workflows to digitalize the faculty's administrative processes.

The document serves as the primary technical reference for the development team, QA testers, project managers, and university stakeholders (secretariat staff and faculty).

1.2. Document Conventions
- Headings: Times New Roman, size 14, bold (per the document template).
- Body text: Arial, size 11.
- "Shall" indicates mandatory behavior; "should" indicates recommended behavior.
- Functional requirements use IDs REQ-XX, non-functional requirements use NFR-XX.
- Bold text highlights key terms when first introduced.

1.3. Intended Audience and Reading Suggestions
Each audience should prioritize sections relevant to their role:
- Registrar and Secretariat Staff (Primary): Section 2.2 (Product Functions) and Section 3.1 (User Interfaces).
- External Visitors and Prospective Students: Section 2.2 and Section 3.1.1 (Public Web Interface).
- Software Developers and Architects: Section 3 (External Interface Requirements) and Section 4 (System Features).
- System Administrators: Section 5.2 (Safety) and Section 5.3 (Security).
- Academic Staff (Professors): Section 2.2 and Section 3.1 for grade entry and communication features.
- QA and Testing Teams: Section 3 for verifiable requirements and Section 6 for testing mandates.

1.4. Product Scope
The product is the Automated Faculty Student Management System (AFSMS / GASF).

Objectives and Benefits
AFSMS automates the faculty's manual administrative workflows. Key benefits:
- Eliminates data redundancy and manual entry errors via client-side validation and centralized database management.
- Reduces processing time for generating official documents (e-Transcript, e-Grade Centralizer).
- Provides 24/7 information access for students and staff via a secure, role-based web portal.

What the System Will Do
AFSMS provides web-based tools for:
- Academic data collection and student record management.
- Advanced document search and filtering.
- Secure electronic document circulation within the secretariat.
- Report generation, exportable in CSV, XML, XLS, and PDF formats.
- Integration with Microsoft Excel and Microsoft Outlook.
- Activity logging, offline backups, and point-in-time database recovery (rollback).

What the System Will Not Do
AFSMS will not:
- Process financial transactions or manage tuition fees.
- Interface with central HR or payroll systems.
- Replace institutional systems outside its academic scope.

2. Overall Description

2.1. Product Perspective
AFSMS is a new, standalone software product. It provides faculty-level academic management but is not a component of any larger institutional system — it will not replace the university's financial or HR systems.

2.1.1. System Interfaces
AFSMS operates independently at the registrar's office level, with minimal interfaces to higher-level university systems. It exchanges data through standardized imports and exports, focusing strictly on faculty-level workflows.

2.1.2. User Interfaces
- Logical characteristics: The interface displays in a web browser with two perspectives (public and private). Screens use tabular layouts for grade registers and summaries, consistent navigation menus, and help/search functions.
- Optimization: The interface uses selection lists (dropdowns) extensively to reduce errors and increase speed.
- Verifiable usability: The system should allow rapid generation of grade summaries after minimal staff training.

2.1.3. Hardware Interfaces
- Server: The physical or virtual server architecture must support concurrent processing.
- Client: No special hardware is required. Any internet-connected device (desktop PC, laptop, or tablet) with an up-to-date web browser is supported.

2.1.4. Software Interfaces
The system integrates with:
- RDBMS: Stores student data and activity logs; supports web-based administration and offline backups.
- Microsoft Excel (.XLS/.CSV): Bidirectional bulk data import and report export.
- Microsoft Outlook (SMTP / Microsoft Graph API): Group messaging and document workflow communications.

2.1.5. Communications Interfaces
All data transfer uses HTTPS over standard ports, encrypting academic and authentication data (SSO) in transit between browser and server.

2.1.6. Memory Constraints
Server-side storage must be scalable to accommodate large volumes of historical data (spanning years of study) and comprehensive operation logs required for the rollback function.

2.1.7. Operations
- Interactive: Grade entry, student queries, and document management during office hours.
- Unattended: Automatic offline database backups (preferably outside peak hours).
- Special: Administrator-triggered rollback and point-in-time recovery for data corruption or major errors.

2.1.8. Site Adaptation Requirements
The system must support faculty-specific configuration before first use, including pre-configured curricula, department types, local grading rules, and academic year structures (grid values).

2.2. Product Functions
AFSMS automates student management and administrative workflows through the following functions:
- Data Collection: Gather curricula, study formations, and student records.
- Data Management: Centralized administration of academic information; data extraction from databases, Excel files, and text files.
- Report Generation: Produce official documents (e-Transcript, e-Grade Centralizer) in CSV, XML, XLS, and PDF.
- Document Workflow Management: Electronic circulation of documents within the registrar's office for information, approvals, or modifications.
- Querying and Search: Locate documents and data by file type, creation date, author, or content, using selection lists for easy navigation.
- Web Portal: Differentiated access via public and private perspectives; registered users can modify data or generate reports based on their privileges.
- Logging and Data Recovery: Record all operations in a log file; support data rollbacks to prevent collection errors.

2.4. Operating Environment
AFSMS uses a client-server architecture and operates in the following environments:

Server Environment
- Hardware: Physical servers in the university data center or virtualized/cloud infrastructure with scalable processing and storage.
- Operating Systems: Linux (Ubuntu Server 22.04/24.04 LTS, CentOS) or Windows Server (2019, 2022).
- Software: An RDBMS supporting complex queries, offline backups, and point-in-time recovery.

Client Environment
- Hardware: Any internet-connected device — desktop PCs, laptops, tablets, or smartphones.
- Operating Systems: Windows (10, 11), macOS, desktop Linux, Android, iOS.
- Browsers: Google Chrome 100+, Mozilla Firefox 100+, Microsoft Edge, Apple Safari.

Compatibility
On registrar workstations, the client must coexist with Microsoft Excel (2016, 2019, 2021, or Microsoft 365) and Microsoft Outlook without memory conflicts, supporting CSV/XLS import/export and institutional communication.

2.5. Design and Implementation Constraints
- Regulatory Policies: The system must comply with EU/Romanian GDPR for personal student data and with the University of Craiova's academic data retention policies.
- Hardware/Memory: The server requires scalable storage for comprehensive activity logs needed by point-in-time recovery.
- Technologies: The backend must use an RDBMS. The frontend must use standard web technologies (HTML/CSS/JS) for cross-platform compatibility with no local client installation.
- Security and Protocols: All data transfers must use HTTPS. Authentication must support institutional SSO.
- Application Interfaces: Export and communication modules must interface with Microsoft Excel (.XLS/.CSV) and Microsoft Outlook.
- Design Conventions: Developers must use selection lists (dropdowns) over free-text fields for standardized data entry. Reports must display in tabular format.
- Audit and Concurrency: The DBMS must support full activity logging (audit) and point-in-time rollback. The system must handle concurrent access during peak exam sessions without data corruption.

The following documentation, organized by user role, will accompany AFSMS.

2.6.1. Delivered Documentation Components
- Registrar & Secretariat Staff Operations Manual:
  - Scope: Complete guide for daily workflows and data entry.
  - Contents: Step-by-step instructions for dropdown-based data collection, electronic document circulation, advanced search filters, Excel import/export, Outlook communications, and generating the e-Transcript and e-Grade Centralizer.

- Professors & Teaching Staff Quick Start Guide:
  - Scope: Concise, visual tutorial for faculty-relevant features.
  - Contents: SSO authentication, locating study formations, entering session grades, and finalizing evaluation records.

- System & Database Administrator Guide:
  - Scope: Technical documentation for IT staff.
  - Contents: Site adaptation procedures (curricula, grid values), user privilege mapping, DBMS management, activity log interpretation, offline backups, and point-in-time recovery (rollback).

- Student Portal Guide (Online Help):
  - Scope: Brief FAQ embedded in the private portal.
  - Contents: Viewing personal schedules, interpreting grades, and requesting administrative documents.

2.6.2. Interactive and In-App Documentation
- Context-Sensitive Help: Embedded tooltips next to complex fields and dropdowns.
- Error Resolution Prompts: Client-side validation errors include resolution suggestions (e.g., "Invalid format: Ensure the Excel file matches the provided .XLS template before importing").

2.6.3. Delivery Formats and Tooling
- Web-Based Documentation: Online help section integrated within the web portal (guides, FAQs, contextual help).
- PDF Manuals: Registrar, Professor, and Administrator guides in PDF for offline access and archiving.
- All documentation will be updated alongside major software releases.

2.6.4. Documentation Standards and Compliance
- ISO/IEC/IEEE 26514:2022: Standard for user documentation design and development.
- WCAG 2.2 (Level AA): HTML5 and PDF documentation must meet accessibility requirements (screen reader compatibility, color contrast, keyboard navigation).
- GDPR Compliance: All sample data and screenshots must use anonymized, synthetic data — no real student PII.

2.7. Assumptions and Dependencies
The following assumed factors and external dependencies influence the requirements in this SRS. If any prove false, the affected requirements will need revision.

2.7.1. Assumptions (Factors Assumed to be True)

Infrastructure and Provisioning:
- Assumed: The university IT department will provide a data center or cloud-hosting environment with adequate availability during academic periods and sufficient bandwidth for exam-season traffic.
- If false: The SRS must add requirements for an external cloud architecture (e.g., AWS/Azure) and load-balancing.

Centralized SSO Availability:
- Assumed: The university SSO subsystem provides a stable authentication endpoint (SAML 2.0 or OAuth 2.0).
- If false: The SRS must add requirements for secure password hashing, credential storage, account recovery, and MFA.

Browser Standards and Connectivity:
- Assumed: All users have reliable internet and devices capable of rendering HTML5 and ES6 JavaScript.
- If false: The SRS must add requirements for offline synchronization and legacy browser polyfills.

Open-Source / Commercial Libraries:
- Assumed: The team has unrestricted access to libraries for generating CSV, XML, XLS, and PDF exports.
- If false: Export requirements must be scaled down, or the budget/timeline expanded.

Administrative User Training:
- Assumed: Registrar staff will complete a mandatory 1-hour training session before deployment.
- If false: The 3-minute usability benchmark cannot be fairly evaluated at UAT and must be removed.

2.7.2. Dependencies (External Factors Relied Upon)

Microsoft Office Ecosystem:
The export, import, and communication modules depend on Microsoft Excel and Microsoft Outlook being available and licensed on registrar machines. If Microsoft deprecates interoperability protocols or changes .XLS parsing rules, integration modules will need refactoring.

Legacy Data Availability (Site Adaptation):
Launch depends on the secretariat providing cleansed, structured legacy data (historical records, curricula, past grid values) for database seeding. Delays or formatting errors in this data will block deployment.

Regulatory Environment:
Logging, data retention, and rollback features depend on current EU/Romanian GDPR legislation and Romanian Ministry of Education archiving mandates. Legislative changes will force revisions to database architecture and data-pruning logic.

Institutional Network Security:
Automated notifications and user authentication depend on the university IT department whitelisting the application servers for outbound SMTP traffic and inbound SSO callbacks.


3. External Interface Requirements

3.1. User Interfaces
AFSMS provides a web-based GUI accessible through modern browsers, organized into:
- Public Portal: Unauthenticated, read-only access.
- Private Portal: Authenticated via university SSO with role-based access control (registrar staff, professors, students, administrators).

3.1.1. Logical Characteristics and Screen Organization
- Layout: A persistent two-pane design — a fixed, role-based navigation sidebar on the left and a dynamic workspace on the right. Breadcrumb navigation appears at the top of the workspace.
- Screen Formats: All academic records (grade registers, centralizers) use a standard tabular format mirroring MS Excel for familiarity.
- Standardized Elements: Every authenticated screen includes a persistent top header with a universal search bar, user profile/logout controls, and a "Help" icon for context-sensitive assistance.
- Keyboard Shortcuts: The system supports grid-navigation shortcuts (Tab to cycle columns, Enter to save the row and advance) to optimize repetitive data entry.

3.1.2. UI Optimization and Verifiable Usability
- Selection-Only Mandate: The UI must use searchable dropdown menus (e.g., Select2/React-Select) for all relational data inputs (study formations, subjects, academic years). Free-text fields are allowed only when strictly required for non-standardized descriptions.
- Error Message Standards: Brief, non-technical error messages appear adjacent to the offending input field (e.g., "Grade must be an integer between 1 and 10"). System-level errors include a "More Info" toggle for technical details.
- Verifiable Usability Requirement: A secretariat staff member shall compile and export a complete semester e-Grade Centralizer for a 30-student group in under 3 minutes, after no more than 1 hour of basic training.

3.2. Hardware Interfaces
AFSMS is a browser-based client-server system. Its hardware interfaces are defined by the server infrastructure and end-user devices (per IEEE 5.2.1.3 and IEEE 5.3.1).

3.2.1. Server-Side Hardware Interface
- Deployment: Deployable on physical (on-premises) or virtualized servers supporting multi-threaded concurrent processing.
- Storage: Scalable primary and secondary storage for the operational database and long-term activity log retention (auditing and point-in-time recovery).
- Network: The server exposes the application over standard network interfaces (Ethernet/Wi-Fi via the institutional network).

3.2.2. Client-Side Hardware Interface
- Supported Devices: Any internet-connected device with a modern browser — desktop PCs, laptops, tablets, and smartphones.
- Interaction: Full-screen graphical UI within the browser; no legacy terminal interactions.
- Peripherals: No specialized hardware required (no barcode scanners or smart-card readers). Standard input only: keyboard and mouse on PCs/laptops; touch on mobile devices.

3.2.3. Hardware-Related Configuration Characteristics
- Ports: Inbound TCP port 443 (HTTPS) required. All other service ports (e.g., database administration) restricted to internal network interfaces.

3.2.4. Logical Characteristics of Hardware Interactions
- Network Tolerance: The server must handle up to 10,000 simultaneous TCP connections without packet loss during peak exam periods.
- Network Latency: Server routing must support round-trip latency under 100ms.
- Storage I/O: Database drives must deliver sufficient IOPS to process offline backups without locking the database or timing out user sessions.

3.3. Software Interfaces
This section defines the logical connections between AFSMS and external software components.

3.3.1. Interfaces with External Applications

Data Persistence Interface (DBMS)
- Purpose: Persistent storage and retrieval of academic data and audit logs; supports point-in-time recovery and offline backups.
- Inbound: Query results for student records, curricula, grades, document metadata, user roles, and audit entries.
- Outbound: CUD (Create, Update, Delete) transactions for academic entities; append-only log events for all operations.
- Format: Relational records via SQL (JDBC/ODBC), including rollback recovery commands.

Import and Export Interface (Microsoft Excel)
- Inbound: User-provided structured files (student lists, curricula) validated by the UI layer.
- Outbound: Report exports and tabular datasets (e.g., e-Grade Centralizer).
- Format: .XLSX, .XLS, .CSV, and .XML files. Column definitions and templates are documented.

Communication and Workflow Interface (Microsoft Outlook)
- Outbound: Composed emails (recipients, subject, body) with optional attachments (PDF reports).
- Inbound: Basic delivery status (if supported by the API); primarily send-initiated.
- Format: Standard email fields via Microsoft Graph API or SMTP; attachments in PDF/XLS/CSV.

Authentication Interface (University SSO)
- Inbound: Authentication assertions/tokens and unique user IDs after login.
- Outbound: Authentication requests and redirects to the SSO endpoint.
- Format: SAML assertions or OAuth2/OIDC tokens (implementation-dependent).

Report Rendering Interface (PDF-Gen)
- Inbound: Compiled binary file for download or email attachment.
- Outbound: HTML or tabular data templates pushed to the rendering engine.
- Format: .PDF files conforming to ISO 32000.

3.3.2. Shared Data and Constraints
- Data Sharing: The system exports structured academic datasets in .CSV, .XML, .XLS, and .PDF.
- Authentication Restriction: Authentication is constrained to the institutional SSO. A standalone credential database is prohibited by institutional security policy.
- Transaction Integrity: The DBMS must enforce strict transaction logging for every operation to guarantee point-in-time recovery (rollback).

3.4.1. Network and Web Server Communications
- Source/Destination: All primary interactions occur over WAN or LAN, from the client web browser to the AFSMS web server.
- Protocols & Standards:

3.4.2. Message Content, Formatting, and Commands
- Client to Server: HTTP requests (GET/POST) carrying form parameters, filter criteria, and search queries. File uploads use multipart/form-data.
- Server to Client: HTTP responses with tabular data and validation results, primarily as JSON payloads or binary file downloads.
- The UI layer must interpret standard HTTP status codes to display user-friendly error messages.

3.4.3. Timing, Transfer Rates, and Synchronization
- Synchronous: General data entry and portal navigation provide real-time UI feedback.
- Asynchronous: Resource-intensive operations (bulk PDF generation, mass email) execute asynchronously with a polling mechanism or progress indicator to prevent UI blocking.
- Transfer Rates: Dependent on institutional bandwidth. Exact timeout thresholds and minimum transfer rates are TBD (detailed design phase), but must support the 3-minute usability benchmark (Section 2.1.2).

3.4.4. E-mail and Document Workflow Communications
- Source/Destination: Initiated by the AFSMS server; delivered to the institutional Microsoft Exchange/Outlook server and ultimately to user inboxes.
- Protocols: Bulk group messages use the Microsoft Graph API (RESTful HTTPS) or SMTP.
- Formatting:
  - Emails use the MIME standard (plain text and rich HTML).
  - Workflow documents are transmitted as secure attachments in .pdf, .xls, or .csv format.

3.4.5. Data Export and Download Mechanisms
- Protocols: Exports download through the user's browser over HTTPS. No FTP/SFTP is required or permitted for end-user exports.
- Formats:
  - CSV (Comma-Separated Values)
  - XML (Extensible Markup Language)
  - XLS/XLSX (Microsoft Excel Binary/OOXML)
  - PDF (Portable Document Format)


4. System Features
This section organizes AFSMS functional requirements by major system feature.

4.1. Web Portal and Access Management

4.1.1. Description and Priority
The web portal provides SSO-based authentication, user registration with an initial predefined rights set, and role-based access control across a public (unauthenticated) and private (authenticated) perspective.

Priority: High.

4.1.2 Functional Requirements
REQ-AFSMS-01: The system shall provide a public portal perspective accessible without authentication.
REQ-AFSMS-02: The system shall provide a private portal perspective accessible only to authenticated users.
REQ-AFSMS-03: The user interface shall be provided as a browser-based web interface.
REQ-AFSMS-04: The system shall display data and query results in tabular form within the web page.
REQ-AFSMS-05: The system shall enforce differentiated access to data based on user roles/rights in the private portal.
REQ-AFSMS-06: The web interface shall be secure and shall provide differentiated access for different user roles.
REQ-AFSMS-07: The system shall allow a user to register through the web interface.
REQ-AFSMS-08: The system shall assign each newly registered user an initial predefined set of access rights.
REQ-AFSMS-09: The system shall allow an authorized administrator to customize user access rights after registration.
REQ-AFSMS-10: If registration input data is invalid, the system shall reject the request and display an error message.

Appendix A: Glossary

AFSMS (Automated Faculty Student Management System): The software product specified here, designed to automate administration of student records, curricula, and document circulation within a university faculty.

e-Registrul matricol (e-Transcript): A secure electronic document recording a student's full academic history — all completed disciplines, contact hours, and ECTS credits.

e-Centralizatorul de note (e-Grade Centralizer): A standardized electronic report aggregating student exam results within an academic year, organized by specialization and study formation.

Managementul fluxului de documente (Document Workflow Management): The system capability for electronic routing, tracking, and storage of administrative documents for information, approval, or modification.

Rollback: A data integrity function enabling the database to revert to a previously verified stable state, neutralizing data corruption or collection errors.


Appendix B: Analysis Models

This appendix provides logical models illustrating AFSMS functional requirements and data relationships.

B.1. High-Level System Context Diagram
The Context Diagram defines the AFSMS boundary and its external entities:
- AFSMS Core System: Processes academic requests, report generation, and document workflows.
- External Entities:
  - Registrar/Secretariat Staff: Provide student data, curricula updates, and document routing.
  - Teaching Staff (Professors): Provide examination results and grades.
  - Students: Receive academic standings, schedules, and grade reports.
  - University SSO Subsystem: Provides authentication tokens.
  - Microsoft Ecosystem: Facilitates Excel import/export and Outlook communications.

B.2. Data Flow Diagram (DFD Level 0)
Primary data transformations:
- Data Collection: Accepts raw data from external files (Excel, text) and databases, applies transformations and filters for system integration.
- Report Processing: Transforms stored academic records (grades, ECTS, hours) into standardized formats (e-Grade Centralizer, e-Transcript).
- Audit & Log Generation: Generates a log entry for every interaction, identifying the user and data modification for safety and rollback.
- Workflow Management: Directs electronic document circulation within the secretariat for approval or modification.

B.3. Entity-Relationship Model (Conceptual)
Fundamental data structures and relationships:
- STUDENT: Uniquely identified by personal details; enrolled in a STUDY FORMATION.
- CURRICULUM (Plan de învățământ): Defines DISCIPLINES, ECTS credits, and contact hours for a specialization.
- GRADE: Links a STUDENT to a DISCIPLINE for a specific academic year.
- USER ACCOUNT: Mapped to ROLES and PRIVILEGES for differentiated private portal access.
- LOG ENTRY: Append-only record linked to a USER ACCOUNT and an OPERATION.

B.4. High-Level System States
AFSMS has two primary logical states:
- Public Portal State: Default state where visitors browse unauthenticated data (curricula, announcements).
- Private Portal State: Activated on successful SSO authentication. Provides differentiated services (Data Management, Reporting, Rollback) based on the user's role (Student, Professor, Registrar).
